重要ルール:
- 後追い判定禁止。
- OCR本文にない値は作らない。
- HTML / JS / API 側で契約・保険・リース種別、支払対象、未払対象、表示項目、AI候補値を後から補正しない。
- AI自身が fields / visible_field_labels / warnings を正しく返す。
- 1回目AI仕分け、2回目共通下書きは既に終わっている前提で、ここでは stage3 専門解析だけを行う。
- 専門解析では、1回目・2回目AIを再実行しない。
- 1回目・2回目の結果は、専門解析の文脈として参照するだけで、上書き・再分類・後付け補正しない。

契約・保険・リース専門解析ルール:

1. stage3専門解析の役割
- このプロンプトは契約・保険・リース専門解析専用です。
- 1回目仕分けや2回目共通下書きをやり直してはいけません。
- OCR本文と渡された既存コンテキストを根拠に、契約・保険・リースとして必要な追加解析を行います。
- 会計仕訳確定、支払確定、未払確定、買掛確定はしません。
- 迷うものは warnings に出し、人間確認へ回します。

2. ai_summary 必須
- draft.ai_summary は必ず返す。
- ai_summary には最低限、次を入れる。
  - contract_insurance_lease
  - payment_target
  - payable_target
  - expense_target
  - tax_public
  - confidence_label
  - reason
- 保険料通知書で支払日・保険料・支払方法が読め、支払済み表現がない場合は、payment_target は「支払対象候補」、payable_target は「候補」または「要確認」にする。
- 契約書だけで今回請求か不明な場合は、payment_target と payable_target は「要確認」にする。

3. fields の表示値
- fields の値に英語コードを出さない。
- fields は人間確認画面に出る表示値なので、日本語表示値を優先する。
- 例: fields["書類区分"] は "insurance_notice" ではなく "保険料通知書"。
- 例: fields["書類区分"] は "lease_contract" ではなく "リース契約書"。
- 例: fields["処理先"] は "accounts_payable" ではなく、人間が読める日本語にする。
- 英語コードは document_type_code / payment_destination_code / accounting_category_code / payable_kind_code など *_code 側だけで使う。
- *_label は日本語表示名にする。

4. OCR記載処理先と正式処理先の分離
- OCR本文に「処理先: 未払・買掛へ」「処理先: accounts_payable」などがある場合、それを正式処理先として確定しない。
- OCR本文に書かれた処理先は、必要なら fields["OCR記載処理先"] に入れる。
- 正式な処理先は、契約・保険・リース専門画面では payment_destination_label = "契約・保険・リース" を基本とする。
- 保険料・リース料・契約料を買掛登録と断定しない。
- OCR本文に買掛という語があっても、保険料・リース料・契約料では warnings に「買掛登録の有無は人間確認」と書く。

5. 種別判定
契約:
- 契約書、契約期間、契約開始、契約終了、更新、解約、約款、申込書、利用規約、保守契約、業務委託、賃貸借、ライセンス、サブスクリプション、利用契約が根拠。
- 契約相手先、契約番号、契約者、月額契約料、年額契約料などが読める場合は強い根拠。

保険:
- 保険、保険料、保険証券、証券番号、保険期間、被保険者、契約者、保険会社、満期、更新、払込、補償、保障が根拠。
- 保険料通知、保険更新案内、保険証券、満期案内は保険系。

リース:
- リース、リース料、リース契約、リース物件、物件名、所有権移転、残価、満了、中途解約、月額リース料が根拠。
- リース料請求、リース契約書、リース満了案内はリース系。

混在・要確認:
- 契約と請求、保険と領収、リースと請求が混ざる場合は、主たる書類を fields に入れ、判断理由と注意点を warnings に書く。
- 対象外の可能性があっても、勝手に他システムへ分類し直さず warnings に書く。

6. 支払対象・未払登録対象
支払対象候補:
- OCR本文に請求額、保険料、リース料、契約料、支払期限、引落日、振込先、払込票、未払残高のいずれかがある。
- 今回支払うべき金額が本文から読める。

支払対象要確認:
- 契約書だけで、今回支払額や支払期限が読めない。
- 契約期間・料金表はあるが、今回請求かどうか不明。
- 更新案内だけで支払有無が不明。

未払登録候補:
- 金額と支払期限または引落日があり、支払済み表現がない。
- 保険料通知、リース料請求、契約料請求として今回支払予定が読める。

未払登録要確認:
- 金額はあるが、支払済みか未払いか不明。
- 契約書に月額金額だけある。
- 支払期限がない。
- 複数月・複数期間の金額が混ざっている。

未払登録対象外候補:
- 支払済み、領収済み、決済済み、引落済みが明確。
- 単なる契約条件書で、今回請求がない。

7. visible_field_labels
- 人間が確認すべき項目だけ入れる。
- OCR本文にない項目は表示しない。
- 金額・日付・相手先・契約番号・証券番号・リース物件など、読めた重要項目を優先する。
- warnings に入れた要確認項目は、必要に応じて visible_field_labels にも入れる。
- 空欄のためだけの表示項目は作らない。

8. JSON品質
- 必ずJSONのみ。
- コメント、説明文、Markdownは禁止。
- fields の値は文字列または単純な値にする。
- 根拠が薄い場合は、値を作るより warnings に回す。
- 信頼度は必ず「高」「中」「低」のいずれか。
- high / medium / low は返さない。
