【発行日の基本定義】

発行日とは、その書類自体が作成または発行された日である。

書類に記載されているすべての日付の中から、
書類そのものの作成日・発行日として意味を持つ日付だけを発行日として抽出すること。

契約、取引、支払、利用、納付などの内容に関する日付を、
書類の発行日の代わりに使用してはいけない。

日付の判断は、固定的な文字置換やプログラム側の後付け補正ではなく、
OCR本文、項目名、周辺文章、書類種別、発行元、言語、国・地域を読み、
AIが書類全体の意味から判断すること。


【必須出力】

発行日を確定できた場合は、次の両方へ同じ値を返すこと。

- JSON最上位の issue_date
- fields の「発行日」

値はDB保存用のISO形式 YYYY-MM-DD とする。

正しい例:

{
  "issue_date": "2026-07-07",
  "fields": {
    "発行日": "2026-07-07"
  }
}

発行日を確定できない場合は推測せず、空文字を返すこと。

{
  "issue_date": "",
  "fields": {
    "発行日": ""
  }
}

issue_dateおよびfields.発行日には、
日本語、英語、中国語などの原文日付をそのまま返さず、
確定できた場合だけYYYY-MM-DDへ統一して返すこと。


【発行日に相当する表記】

書類自体の作成・発行を意味すると判断できる場合、次の表記を発行日の候補とする。

日本語:
- 発行日
- 発付日
- 作成日
- 文書作成日
- 通知発行日
- 明細発行日
- 請求書発行日
- 請求日
- 通知日

英語:
- Issue Date
- Date Issued
- Issued On
- Document Date
- Statement Date
- Billing Date
- Invoice Date
- Notice Date

中国語:

- 发行日期
- 出具日期
- 開票日期
- 开票日期
- 通知日期
- 製表日期
- 制表日期

ただし、項目名だけで機械的に判断せず、
その日付が書類自体の作成・発行日であるかを周辺文脈から確認すること。


【書類種別ごとの考え方】

請求書:
- 請求書の発行日、請求日、Invoice Dateは、書類の発行日として扱ってよい。
- 支払期限、Due Date、振込期限は発行日ではない。

領収書・レシート:
- 領収書の発行日、領収日、Receipt Dateを発行日として扱う。
- 店頭レシートなど、取引と同時に発行される書類では、取引日時が書類の発行日時を兼ねると明確に判断できる場合に限り発行日としてよい。
- 単なる利用日やサービス提供日は自動的に発行日にしない。

カード利用明細:
- Statement Date、Billing Date、明細発行日、請求書発行日など、明細書全体の作成・発行日を発行日とする。
- 各明細行のカード利用日を発行日にしてはいけない。
- 締日、支払日、引落日、支払期限を発行日にしてはいけない。
- 明細書全体の発行相当日が記載されていなければ空文字とする。

納税通知書・納付通知書:
- 通知書自体の発行日、発付日、通知日、作成日が明確な場合は発行日とする。
- 納期限、納付期限、課税期間、対象年度、賦課期日を発行日にしてはいけない。

納付書・払込票:
- 納付書自体の発行日または作成日が明記されている場合だけ発行日とする。
- 納期限、支払期限、コンビニ取扱期限などを発行日にしてはいけない。
- 汎用納付書や納税者自身が作成する書類で発行日が存在しない場合は空文字とする。

契約書・リース契約書・保険契約書:

- 発行日、作成日、文書作成日が明記されている場合は、その日を発行日として最優先する。

- 明確な発行日または作成日がなく、契約書全体の日付として「契約日」または「契約締結日」が記載されている場合は、その日を事実上の発行日として扱ってよい。

- 契約日または契約締結日を発行日として扱う場合は、その日が契約書全体の成立日・締結日を示していることを、項目名、前後の文章、署名欄、当事者欄、書類全体の構造から確認すること。

- 「契約期間」「有効期間」「適用期間」「保険期間」などの期間表記の中にある開始日を、発行日にしてはいけない。

- 「2026年7月1日から2029年6月30日まで」のように、開始日と終了日が一つの期間として記載されている場合、開始日は契約開始日であり発行日ではない。

- 「いつから」「より開始」「開始日」「効力発生日」「適用開始日」など、契約内容が開始する日を発行日にしてはいけない。

- 契約開始日、契約終了日、効力発生日、適用開始日、利用開始日、保険始期日、保険終期日、更新日を発行日の代わりに使用してはいけない。

- 「契約日」と「契約開始日」が別々に記載されている場合は、契約日を発行日候補とし、契約開始日は別項目として扱う。

- 発行日、作成日、契約日、契約締結日のいずれもなく、契約期間や開始日しか記載されていない場合は、発行日は空文字とする。

例1:

契約日: 2026年7月1日

この日が契約書全体の契約日を示しており、
別の発行日または作成日がない場合:

issue_date = 2026-07-01


例2:

契約期間:
2026年7月1日から2029年6月30日まで

この場合:

issue_date = 空文字
契約開始日 = 2026-07-01
契約終了日 = 2029-06-30


例3:

作成日: 2026年6月20日
契約日: 2026年7月1日
契約期間: 2026年7月1日から2029年6月30日まで

この場合:

issue_date = 2026-06-20
契約日 = 2026-07-01
契約開始日 = 2026-07-01
契約終了日 = 2029-06-30


例4:

契約日: 2026年6月20日
契約開始日: 2026年7月1日

別の発行日または作成日がなく、
契約日が契約書全体の締結日を示す場合:

issue_date = 2026-06-20
契約開始日 = 2026-07-01


【契約書の日付判定で禁止すること】

- 契約日を一律に発行日対象外としない
- 契約書全体の契約日と、契約期間の開始日を混同しない
- 契約期間の開始日を契約日と取り違えない
- 発行日または作成日が明記されているのに、契約日で上書きしない
- 契約期間の日付しかないのに、開始日を発行日にしない

【日本国内の日付】

次のような西暦表記を理解すること。

- 2026年7月7日
- 2026/7/7
- 2026-7-7
- 2026.7.7
- ２０２６年７月７日

次の和暦を理解し、西暦へ変換すること。

- 令和
- 平成
- 昭和
- 大正
- 明治
- 元年
- R、H、S、T、Mなど、文書全体から元号と判断できる略記

例:

- 令和8年7月7日 → 2026-07-07
- 令和元年5月1日 → 2019-05-01
- R8.7.7 → 令和表記と確認できる場合のみ2026-07-07

元号の期間と矛盾する日付を勝手に補正してはいけない。
元号や年月日を確定できない場合は空文字とする。


【英語圏の日付】

英語の月名表記を理解すること。

- July 7, 2026
- 7 July 2026
- Jul 7, 2026
- 7 Jul 2026

数値だけの日付は、次の情報から国・地域と日付順を判断すること。

- 発行元の国や住所
- 書類言語
- 通貨
- 税番号
- 電話番号
- 郵便番号
- 会社所在地
- 周辺の日付表記
- 英語の綴りや書式

例:

- 米国書類の03/04/2026はMarch 4, 2026の可能性
- 英国書類の03/04/2026は3 April 2026の可能性

月日順を確定できない場合は、推測せず空文字とする。


【中国語圏の日付】

次のような表記を理解すること。

- 2026年7月7日
- 2026年07月07日
- 二〇二六年七月七日
- 簡体字
- 繁体字

台湾書類では、書類全体から台湾の民国紀年と確認できる場合に限り、
西暦へ変換すること。

例:

- 民國115年7月7日 → 2026-07-07

単に「115年」と記載されているだけで、
台湾書類と確認できない場合は民国紀年として扱わないこと。

書類内に複数の日付がある場合は、それぞれの日付の役割を区別すること。

次の日付を発行日に転用してはいけない。

- 支払期限
- 納期限
- 納付期限
- 引落日
- 支払予定日
- 締日
- 利用日
- 取引日

- 課税期間
- 会計期間
- 契約開始日
- 契約終了日
- 効力発生日
- 署名日
- 押印日
- 更新日
- OCR実行日
- ファイル作成日
- ファイル名の日付
- 現在日

ただし、領収書や店頭レシートのように、
取引と同時に書類が発行されたと書類の性質および文脈から明確に判断できる場合は、
取引日を発行日として扱ってよい。

この例外を契約書、カード明細、納付書などへ広げてはいけない。


【OCR誤読への対応】

次の表記揺れは、書類全体から合理的に判断すること。

- 全角数字
- 文字間の空白
- 改行
- 句読点
- 区切り文字
- 軽微なOCR誤読

ただし、年・月・日の一部が読めない場合や、
複数候補を絞れない場合は、推測して日付を完成させず空文字とすること。


【禁止事項】

- 書類内にある日付を何でも発行日にしない
- 現在日から発行日を推測しない
- OCR実行日を発行日にしない
- ファイル作成日を発行日にしない
- ファイル名の日付だけで確定しない
- 支払期限や納期限を発行日にしない
- 契約期間の開始日、契約開始日、契約終了日を発行日にしない
- カードの各利用日を明細書の発行日にしない
- 曖昧な米国式・英国式の日付を根拠なく決めない
- 和暦を単純な文字置換だけで処理しない
- 台湾と確認できない年を民国紀年として扱わない
- AIが確定していない日付をプログラム側で後付けする前提にしない
- issue_dateへ原文の日付表記を返さない

YYYY-MM-DDの正式値を返すこと。
