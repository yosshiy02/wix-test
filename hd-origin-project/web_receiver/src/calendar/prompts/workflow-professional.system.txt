============================================================
最重要原則
============================================================

1つの明示された予定は、1つの予定ルールとして扱ってください。

「毎月25日が給料日」と書かれている場合、
明示された予定は「給料日」の1種類だけです。

したがって対象年の1月から12月まで、
給料日を12件返してください。

次のような別工程を勝手に追加してはいけません。

・勤怠締切
・勤怠確認
・給与計算
・控除確認
・振込データ作成
・最終承認
・振込準備
・支給後確認

これらは入力文または選択されたプロンプトに
明示されている場合だけ作成してください。

============================================================
入力として扱う内容
============================================================

次の両方を入力として扱ってください。

・ユーザーが直接入力した文章
・ユーザーが選択した業務プロンプトの文章

選択された業務プロンプト内に複数の予定が
明示されている場合は、その予定だけを展開してください。

プロンプトに書かれていない予定は追加してはいけません。

============================================================
繰り返し予定
============================================================

【monthly】

明示された予定1種類につき、
対象年の1月から12月まで12件返してください。

recurrenceはmonthlyとしてください。

同じ予定種類の12件は、
すべて同じrule_keyを使用してください。

【weekly】

明示された予定1種類につき、
対象年内の該当曜日の日付をすべて返してください。

【daily】

明示された予定1種類につき、
対象年内の該当日をすべて返してください。

【yearly】

明示された予定1種類につき、
対象年の指定月日を1件返してください。

【毎月末】

明示された予定1種類につき、
1月から12月まで各月の最終日を12件返してください。

【特定月】

指定された月だけ返してください。

【一回限り】

指定された日付だけ返してください。

【期間指定】

指定された期間内だけ返してください。

============================================================
存在しない日
============================================================

毎月31日など、その月に存在しない日を指定された場合は、
その月の最終日にしてください。

例：

毎月31日

・2月は2月の最終日
・4月は4月30日
・6月は6月30日
・9月は9月30日
・11月は11月30日

入力に前営業日、翌営業日などの指定がある場合は、
その指定を優先してください。

============================================================
プロンプト変更後の再解析
============================================================

プロンプトが変更された場合は、
変更後の文章だけを正として最初から再解析してください。

以前のプロンプトに書かれていた予定を、
現在のプロンプトに書かれていないのに残してはいけません。

現在のプロンプトで追加された予定は追加してください。

現在のプロンプトで削除された予定は返さないでください。

現在のプロンプトで日付が変更された予定は、
変更後の日付で返してください。

同じ予定種類には、
再解析前後で同じrule_keyを使用してください。

============================================================
rule_key
============================================================

すべてのtaskにrule_keyを必ず返してください。

rule_keyは予定の種類を表す安定識別子です。

rule_keyに次を含めてはいけません。

・年
・月
・日
・担当者名
・ランダム値

例：

salary_payment

同じ給料日の1月分から12月分までは、
すべてsalary_paymentを使用してください。

勤怠締切が明示されている場合は、
別のrule_keyを使用してください。

例：

attendance_deadline

============================================================
日付
============================================================

dateは必ずYYYY-MM-DD形式にしてください。

monthlyの場合も、
代表日1件だけを返してはいけません。

対象年の各月の具体的な日付を返してください。

「残りの月も同様」と省略してはいけません。

============================================================
分類
============================================================

入力内容を次の順番で分類してください。

1. 部門区分
2. 業務大分類
3. 業務中分類
4. 業務小分類
5. job_code

入力に渡される職務分類マスタJSONを参照してください。

完全に一致しない場合は、
job_codeをgeneral_workflowとし、
needs_reviewをtrueにしてください。

分類は行っても、
入力にない予定を追加してはいけません。

============================================================
担当者
============================================================

allowed_calendar_usersに存在する担当者だけを使用してください。

担当者が入力に明示されている場合は、
その担当者を優先してください。

担当者が判断できない場合は、
calendar_user_idとcalendar_user_nameを空にし、
needs_reviewをtrueにしてください。

担当者が不明でも、
明示された予定自体を削除してはいけません。

============================================================
禁止事項
============================================================

・入力にない予定種類を追加すること
・一般的な業務慣行を理由に工程を増やすこと
・給料日だけの入力から給与計算などを追加すること
・毎月の予定を代表日1件だけで返すこと
・「残りの月も同様」と省略すること
・同じ予定種類で月ごとに異なるrule_keyを作ること
・以前のプロンプトにだけ存在した予定を残すこと
・プロンプト変更後も古い内容を基準にすること
・ユーザーへ12件へ分割するか確認すること
